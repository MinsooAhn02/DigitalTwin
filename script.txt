Hello everyone.
For this mid-check, we will show what our system can do now, what problems remain, and what
we plan to improve next.

We load CCTV locations for the current map area via the ITS OpenAPI and place markers on
the map. Clicking a marker opens a floating panel that plays the live HLS stream using HLS.js.

The panel has two tabs — live video and YOLO detection. On the YOLO tab, the browser
captures frames from the video, sends them to the backend via WebSocket, and displays the
annotated result. We took this approach because the ITS HLS URL contains an expiring token,
making direct OpenCV access on the backend unreliable.

In the detection pipeline, YOLOv8 identifies cars, buses, trucks, and motorcycles, and
ByteTrack maintains consistent IDs across frames. Results are reflected on the map and sidebar.

We also fixed a frame-ordering issue where delayed frames could appear after newer ones.
The frontend now holds the next frame until the previous response arrives, preventing
out-of-order rendering.

For performance, CPU inference took over 500ms per frame — far too slow for real-time use.
By adding CUDA GPU acceleration and converting the model to a TensorRT FP16 engine, we
reduced that to around 50ms in practice, roughly a 10x improvement over CPU.

---

Now, what still needs improvement.

The first issue is measurement accuracy. Speed is currently calculated using a fixed frame rate assumption,
but the actual detection interval varies at runtime, so the values are not precise.

The second issue is camera angle. Public CCTV cameras are mounted at various fixed angles,
and when the angle is poor, our pixel-to-GPS mapping becomes unreliable — which also makes speed and position estimates less accurate.

The third issue is detection scope. Since we are using street-facing CCTV,
the field of view often includes parking lots, sidewalks, or side streets,
so parked or off-road vehicles can be picked up as moving traffic and add noise to the data.

The fourth issue is GPU thermal throttling. Under sustained load, inference time gradually
increases from around 20ms to over 100ms as the laptop GPU heats up and reduces its clock
speed. On a server-grade GPU with proper cooling, this degradation would not occur.

The fifth issue is the WebSocket pipeline structure. The browser captures a frame, encodes
it as JPEG, sends it to the backend, and waits for the annotated result before sending the
next frame. This one-at-a-time round trip means the full pipeline latency — capture, transfer,
inference, and return — becomes the bottleneck, limiting effective throughput.

These are the main limitations we are aware of at this stage.

For next steps, we have a plan to address each of these. For speed accuracy, we will replace
the fixed FPS assumption with actual per-frame timestamps, so that speed is calculated using
the real time elapsed between detections rather than an assumed interval.
For camera angle, we plan to apply homography calibration — by marking a few known real-world points in the
camera image, such as lane markings, we can compute a transformation matrix that corrects
for perspective distortion and gives more accurate pixel-to-GPS mapping.
For detection scope, we will apply ROI masking to exclude irrelevant areas like parking lots and sidewalks from
the detection zone. And for the thermal throttling issue, we plan to implement adaptive frame
skipping under high load, and ultimately move toward server deployment where sustained GPU
performance is more stable.
For the pipeline bottleneck, we are currently researching alternative approaches that could
reduce the round-trip overhead and improve throughput.

Q&A

Q: What is your overall tech stack?
A: The frontend is built with React and MapLibre GL for the map. The backend is FastAPI
running in Python. For detection we use YOLOv8 from Ultralytics with a TensorRT FP16 engine
for GPU acceleration, and supervision for ByteTrack multi-object tracking. The CCTV stream
is played in the browser using HLS.js, and frames are sent to the backend via WebSocket.

Q: Why did you play the HLS stream in the browser instead of just opening it on the backend?
A: The ITS CCTV URLs include an expiring authentication token. By the time OpenCV on the
backend tries to open the stream, the token may have expired, causing the connection to fail.
The browser handles the HLS protocol natively and manages token renewal automatically, so we
capture frames from the browser video element and forward them to the backend instead.

Q: How does the YOLO and ByteTrack pipeline work together?
A: YOLO runs on each frame and returns bounding boxes with class labels and confidence scores.
ByteTrack then takes those detections and assigns consistent IDs across frames by matching
new detections to previously tracked objects using motion prediction. This means even if a
vehicle is briefly occluded or missed in one frame, ByteTrack can re-associate it with the
same ID when it reappears.

Q: How do you convert pixel positions to GPS coordinates?
A: Currently we use a simple linear approximation based on the camera's known GPS position.
The pixel offset from the center of the frame is scaled by a fixed factor to estimate the
real-world offset in latitude and longitude. It is a rough estimate and becomes inaccurate
when the camera angle is steep, which is why homography calibration is one of our planned
improvements.

Q: What is TensorRT and why did you use it?
A: TensorRT is NVIDIA's inference optimization library. It takes a trained model and compiles
it into an engine optimized for a specific GPU, applying FP16 precision to reduce memory and
computation. In our case it reduced backend-only inference from around 500ms on CPU down to
roughly 50ms on GPU, making the live pipeline practical for demo use.
