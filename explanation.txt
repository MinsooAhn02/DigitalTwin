  최종 업데이트: 2026-05-27 (양방향 도로 중심 보정, 인접 링크 확장, 곡선 polygon)

이 문서는 프로젝트를 처음 접하는 사람도 전체 구조와 동작 방식을
이해할 수 있도록 작성된 기술 설명서입니다.


================================================================
1. 프로젝트 개요
================================================================

실제 도로의 CCTV 영상을 AI로 분석하여, 지도 위에 차량 위치·속도·
교통 흐름을 실시간으로 시각화하는 교통 디지털 트윈 시스템입니다.

핵심 기능:
  - ITS(국가교통정보센터) OpenAPI에서 실제 CCTV HLS 스트림 수신
  - YOLOv8 + BoxMOT 기반 차량 탐지 및 다중 객체 추적
  - 픽셀 좌표 → GPS 좌표 변환 (4점 호모그래피 보정)
  - deck.gl 지도 위 차량 마커, 속도, LOS, 경보 실시간 표시
  - ROI(관심 영역) 설정, 카메라 방향 보정 UI
  - 국가 노드링크 DB 연동: 도로명·제한속도·bearing 자동 조회
    → 카메라 GPS를 도로 shape_pts 폴리라인에 투영(snap)하여 도로 중심선 좌표 계산
    → snap 좌표를 transformer GPS 기준점 및 FOV polygon 원점으로 사용
  - CCTV 이름에서 도로명·방향 파싱 → 지도 FOV 정렬, 과속 기준 자동 적용
    → 대괄호 형식([국도 1호선])과 평문 형식(국도1호선) 모두 지원
  - 노드링크 역방향 링크 감지로 편도/양방향 자동 판별
    → 양방향: lanes × 2 × 차선폭, 편도: lanes × 1 × 차선폭
  - 양방향 도로 중심 보정: 역방향 carriageway snap 평균 → 실제 도로 폭 중심에 polygon 배치
  - 인접 링크 확장: 링크 경계 근방 snap 시 앞뒤 링크 shape_pts 연결 → 짧은 polygon 방지
  - 곡선 FOV polygon: nodelink shape_pts 폴리라인 따라 도로 곡선을 반영하는 polygon 생성
  - 2단계 GPS 변환: pixel→미터→도로 축 분해→road_pts 보간→GPS (직선 호모그래피 대비 곡선 정확도 향상)
  - 소실점(VP) 기반 카메라 시선 방향 자동 판별
    → name_bearing 없을 때만 VP flip 적용 (name_bearing 확정 시 스킵)
  - GPU(CUDA/TensorRT) 추론 자동 선택

3. 기술 스택
================================================================

[ 백엔드 — Python 3.11 ]
  웹 서버       FastAPI + Uvicorn (비동기 REST + WebSocket)
  AI 탐지       ultralytics YOLOv8x / YOLOv8s
                TensorRT FP16 (.engine) — GPU 가속 추론
                torch 2.6.0+cu124 (CUDA 12.4)
  차량 추적     BoxMOT — BotSort / ByteTrack / OcSort / DeepOcSort
                (TRACKER_TIER 환경변수로 선택, 기본 auto)
  영상 처리     OpenCV-Python (cv2), FFmpeg (HLS 디코딩)
  좌표 변환     numpy (homography 행렬 연산)
  HTTP          httpx (비동기 ITS API 호출)
  설정          python-dotenv, pydantic

[ 프론트엔드 — JavaScript (ES2022 + JSX) ]
  빌드          Vite + React 18 (함수형 컴포넌트 + Hooks)
  지도          deck.gl (ScatterplotLayer, TextLayer, PathLayer,
                         PolygonLayer, IconLayer)
                react-map-gl + MapLibre GL (벡터 타일)
                Carto Dark Matter 스타일 테마
  HLS 재생      hls.js (브라우저에서 m3u8 재생)
  차트          Recharts (파이 차트, 레이디얼 바)
  통신          브라우저 네이티브 WebSocket, fetch API
  상태 관리     useState / useReducer / useRef / useCallback / useMemo

[ 외부 서비스 ]
  ITS OpenAPI   https://openapi.its.go.kr:9443/cctvInfo
                현재 지도 뷰포트 bbox 기준 CCTV 목록(위치 + HLS URL) 조회
                HLS URL은 만료 토큰 포함 → 자동 갱신 로직 적용

5. 백엔드 모듈 상세
================================================================

[ config.py ]
  모든 하드코딩 상수를 한 곳에서 관리합니다.
  RUNTIME_PROFILE은 model_setup.py GUI 실행 후 .runtime_profile.json에
  저장되며, 서버 시작 시 이 파일을 읽어 동적으로 오버라이드합니다.

  주요 설정:
    YOLO_MODEL              사용할 모델 파일 (.pt 또는 .engine)
    YOLO_CONF / YOLO_IOU    탐지 임계값 (0.25 / 0.45)
    VEHICLE_CLASSES         탐지 대상 클래스 {2:car, 3:motorcycle, 5:bus, 7:truck}
    TRACKER_TIER            트래커 선택 (auto/cpu/low/medium/high)
    SPEED_LIMIT_KPH         과속 기준 (기본 120)
    MAX_REASONABLE_KPH      이상치 제거 상한 (120 — 도로 제한속도 기준)
    GC_GRACE_FRAMES         트랙 GC 유예 프레임 수 (90 — 약 3초간 ID 보존)
    SPEED_WINDOW_FRAMES     슬라이딩 윈도우 속도 계산 이력 크기 (18프레임)
                            DETECT_INTERVAL=3 기준 실제 샘플 6개 → 회귀 안정성 확보
    YOLO_DETECT_INTERVAL    YOLO 추론 주기 (3 — 3프레임 중 1회만 추론)
    BYTE_TRACK_BUFFER       ByteTrack 트랙 생존 버퍼 (90프레임 = 3초)
    BYTE_TRACK_FPS          ByteTrack 기준 FPS (30)
    BOTTLENECK_DWELL_FRAMES 병목 판정 정지 프레임 (150 = 5초)
    PARKED_FRAMES_THRESHOLD 주차 판정 정지 프레임 (300 = 10초)
    PARKED_POSITION_RADIUS_PX  주차 위치 메모리 반경 (30px)


[ detector.py ]
  VehicleDetector
    - YOLO 모델 로딩: .pt (PyTorch) 또는 .engine (TensorRT FP16) 자동 선택
    - detect(frame): 단순 YOLO 추론 (tracker_id 없음)
    - track(frame): BoxMOT 트래커 연동 추론
        1. YOLO_DETECT_INTERVAL(=3)마다 YOLO predict 실행,
           나머지 프레임은 직전 탐지 결과 재사용 (YOLO 부하 2/3 절감)
        2. CUDA 환경에서 half=True(FP16) 추론 (약 30~50% 속도 향상)
        3. ROI 마스킹 (roi_manager에서 불러온 폴리곤 적용)
        4. tracker.update(bboxes, frame) → track_id 할당/유지 (Kalman 예측 포함)
        5. [cpu/low 티어] _dedup_tracks() → 중복 트랙 제거
        6. [cpu/low 티어] IDStabilizer.update() → ID 안정화
        7. supervision.Detections 객체 반환
    - set_roi(polygon): ROI 폴리곤 업데이트
    - reset_tracker(): 카메라 전환 시 트래커 + IDStabilizer 초기화
    - threading.Lock: live_loop + ws/detect 동시 접근 직렬화

  _dedup_tracks(dets, iou_thr=0.3, dist_thr=90px)
    ByteTrack이 동일 차량에 두 개의 ID를 동시에 출력할 때 제거.
    Kalman 예측(유령 트랙)과 실제 탐지 트랙이 박스가 겹치지 않아도
    중심 거리 90px 이내이면 높은 ID(최신 트랙)를 제거.
    IoU 기준만 쓰면 박스가 안 겹칠 때 유령 트랙을 못 잡는 문제 보완.

  IDStabilizer (cpu/low 티어 전용)
    BoxMOT이 잠깐 놓친 차량에 새 ID를 부여할 때 이전 ID로 복원.
    동작 원리:
      - 매 프레임 이전 활성 트랙을 _lost 딕셔너리로 이동 (매칭 루프 전)
      - 새 raw_id 등장 시 _lost 중 중심 거리 80px 이내 트랙 탐색
      - 찾으면: remap[raw_id] = old_stable_id → 이전 ID 복원
      - 90프레임(3초) 이상 미등장 시 lost 항목 삭제

    2-패스 처리 (중복 ID 방지):
      Pass 1: _remap에 이미 등재된 기존 트랙이 먼저 stable_id를 확보
      Pass 2: 신규 트랙만 _find_lost()로 매칭 시도
      → 새 raw_id가 먼저 처리되어 기존 트랙의 stable_id를 선점하는
        순서 의존 버그(30+ 중복 행) 원천 차단

    stale remap 정리:
      _find_lost() 성공 시 동일 stable_id를 가리키는 _remap 내
      모든 기존 항목을 즉시 제거. 이를 생략하면 트래커가 반복
      ID 재탄생할 때마다 _remap에 동일 stable_id 가리키는 항목이
      누적되어, 해당 프레임에 여러 raw_id가 동시 반환될 때
      30+ 중복 display_id 발생.

    YOLO grace period (YOLO 탐지 누락 시 tracker.update() 건너뜀):
      YOLO_DETECT_INTERVAL=3 환경에서 탐지 결과가 빈 배열일 때
      tracker.update(empty)를 호출하면 ByteTrack/OcSort가 모든 트랙을
      'lost' 처리 → 다음 실제 탐지 시 새 raw_id 부여 → IDStabilizer
      오매칭 → 중복 ID 폭발.
      대응: 연속 2회(_YOLO_MISS_GRACE) 이내 빈 탐지 시 tracker.update
            자체를 건너뛰고 _last_tracks 반환 → tracker 내부 상태 보존
            → 재탐지 시 동일 raw_id 유지 가능.

  VideoStream
    - switch_to(url): OpenCV(FFmpeg 백엔드)로 HLS 스트림 전환
    - read_frame(): 프레임 읽기 (frame_id, ndarray)
    - reconnect(): 스트림 끊김 자동 재연결


[ nodelink.py ]
  국가 표준 노드링크 SQLite DB를 런타임에 조회하는 모듈.
  DB는 scripts/build_nodelink_db.py로 최초 1회 생성 (MOCT_NODE/LINK shapefile 필요).

  노드링크 데이터 특성:
    - 왕복 도로는 방향별로 별도의 단방향 링크 2개로 저장 (F_NODE→T_NODE).
    - 링크는 교차로 사이를 연결하며 교차로에서 끊김 (link boundary).
    - MOCT_LINK.shp의 약 46.9% 링크는 F→T 2점짜리 직선 세그먼트만 보유.
      나머지는 도로 곡률을 반영한 다수의 shape_pts 중간점 포함.
    - shape_pts: SQLite의 JSON 컬럼 [[lat,lon],...] 형태로 전체 폴리라인 저장.

  _snap_to_polyline(lat, lon, pts) → (snap_lat, snap_lon, bearing_deg, dist_m, seg_idx)
    점 (lat, lon)을 폴리라인 pts에 수직 투영.
    bearing_deg: snap이 속한 세그먼트의 로컬 방위각 (전체 링크 F→T 방위와 다를 수 있음).
    seg_idx: snap이 속한 세그먼트 인덱스 (pts[seg_idx]→pts[seg_idx+1]).

  _road_corridor_pts(pts, snap_lat, snap_lon, snap_seg_idx, fwd_m=150, bwd_m=150)
    → (corridor_pts, snap_along_m)
    snap 위치를 중심으로 앞뒤 150m의 도로 중심선 포인트 추출.
    corridor_pts: [[lat,lon],...] F→T 순서
    snap_along_m: corridor_pts[0]에서 snap까지의 누적 거리(m)
    주의: 단일 링크 shape_pts 내에서만 동작 → 링크 경계 근방에서 짧은 결과 반환 가능.

  _extend_pts_with_adjacent(pts, link) → pts (extended)
    링크 경계에서 snap될 때 corridor_pts가 짧아지는 문제를 방지.
    처리:
      1. 현재 링크의 T_NODE를 F_NODE로 갖는 다음 링크를 조회 (앞방향 확장)
      2. 현재 링크의 F_NODE를 T_NODE로 갖는 이전 링크를 조회 (뒷방향 확장)
      각 방향에서 같은 road_name + bearing ±60° 조건의 인접 링크 shape_pts를
      현재 pts 앞/뒤에 이어 붙임 (연결 노드 중복 제거).
    효과: snap이 링크 끝 20m에 있어도 ±150m 범위의 corridor를 충분히 구성.

  _find_reverse_link(primary, links, primary_bearing) → LinkInfo | None
    같은 도로의 반대 방향 carriageway 링크 탐색.
    조건:
      - 같은 road_name (둘 다 이름 있을 때 동일 문자열 필수)
      - bearing이 primary_bearing + 180° ±60° 범위
      - links 후보 중 dist_m 최소인 것 반환
    주의: primary_bearing에는 link["bearing_deg"] (전체 F→T 방위)를 사용.
      local_bearing (shape 세그먼트 로컬 방위)은 곡선 도로에서 40-60° 차이가 있어
      역방향 링크 매칭에 실패할 수 있음.

  _snap_for_link(link, lat, lon) → (snap_lat, snap_lon)
    특정 링크에 대해 (lat, lon)의 snap 좌표 계산.
    shape_pts → F→T 2점 세그먼트 → cx_lat/cx_lon 순으로 폴백.

  get_road_info(lat, lon, road_name_hint=None) → LinkInfo | None
    카메라 GPS 기준 500m 반경 내 가장 가까운 링크 정보 반환.
    road_name_hint 제공 시 ("국도 1호선") 이름 매칭 링크 우선 선택 → 거리 폴백.

  get_road_snap(lat, lon, road_name_hint=None) → dict | None
    get_road_info와 동일한 링크 선택 후, 카메라 GPS를 도로 shape_pts 폴리라인에
    직교 투영하여 도로 중심선 위 snap_lat/snap_lon 계산.

    처리 순서:
      1. _extend_pts_with_adjacent()로 shape_pts 앞뒤 확장
      2. _snap_to_polyline()로 snap 좌표 + seg_idx 계산
      3. _road_corridor_pts()로 ±150m 중심선 추출 → road_pts, snap_along_m
      4. _find_reverse_link()로 역방향 링크 탐색
         → _snap_for_link()로 역방향 snap 계산
         → 두 snap의 거리 2~40m 이내이면 중점 = 실제 도로 중심
         → snap_lat/snap_lon과 road_pts를 같은 횡방향 오프셋만큼 이동
      5. lanes × direction_mult × lane_w = road_width_m 계산

    반환: snap_lat, snap_lon, bearing_deg, road_name, lanes, max_spd,
          road_rank, road_width_m, is_oneway, cam_dist_m,
          road_pts ([[lat,lon],...] 도로 중심선), snap_along_m

    snap 좌표는 transformer GPS 기준점 및 FOV polygon 원점으로 사용.
    road_pts + snap_along_m은 곡선 polygon 및 2단계 GPS 변환에 사용.
    (ITS CCTV GPS는 도로 측면에 위치하므로 도로 중심선과 오프셋이 있음)

  get_nodes_near(lat, lon) → list[NodeInfo]
    보정 모드 캘리브레이션 클릭 시 노드 GPS 스냅핑에 활용.

  LinkInfo 주요 필드:
    link_id, f_node, t_node
    road_name, lanes, max_spd, road_rank, length
    bearing_deg (F_NODE→T_NODE 전체 방위, 0=북 시계방향)
    f_lat, f_lon, t_lat, t_lon (F/T 노드 좌표)
    cx_lat, cx_lon (링크 중심점), dist_m (카메라에서 링크까지 거리)

  DB Path: node-link-data/nodelink.sqlite (R*tree 공간 인덱스)

[ analytics.py ]
  TrafficAnalytics — 매 프레임마다 호출되는 교통 지표 계산 엔진

  입력: VehicleState 리스트 (track_id, class_name, center_px, lat, lon 등)
  출력: FrameAnalytics (vehicles, vehicle_count, avg_speed_kph, los_grade, ...)

  인스턴스 속성 (카메라 전환 시 main.py가 설정):
    speed_limit_kph   과속 기준 — nodelink DB max_spd 우선, 없으면 config 기본값
    road_bearing_deg  도로 진행 방향 (name_bearing ?? road_bearing)
    cam_lat / cam_lon 카메라 GPS
    speed_scale       ITS 비교 자동 보정 계수 (1.0 = 보정 없음, 카메라별 저장)

  처리 순서:
    1. _gc(active): 미감지 트랙 정리 (90프레임 grace period 후 삭제)
    2. LineZone 교차 이벤트(crossed_in_ids, crossed_out_ids)로 direction 갱신
    3. _is_near_parked(): 등록된 주차 위치 30px 이내 → 즉시 is_parked
    4. _speed(): 선형 회귀 기반 속도 계산 (아래 상세 참조)
    5. _dwell_update(): 정지 프레임 누적 → 병목/주차 판정
    6. _project_to_road_axis(): 차량 GPS를 도로 bearing 축에 투영
       - 현재 프레임 차량들의 평균 위치를 기준점으로 사용
       - 횡방향(도로 수직) 흔들림 제거 → 지도 위 차량이 도로선 위에 정렬
    7. 주차 차량 제외 후 통계 집계 (avg_speed, LOS, class_counts)
    8. avg_speed_kph > 0이면 _speed_samples에 기록 (ITS 비교용 5분 이력)

  calibrate_from_its(its_speed_kph):
    ITS 구간속도와 최근 10분 측정 평균을 비교해 speed_scale을 자동 갱신.
      our_avg = 최근 600초 _speed_samples 평균 (window_s=600)
      target  = speed_scale × (ITS / our_avg) = ITS / raw_avg
      EMA 갱신: 미수렴 시 α=0.7(빠른 학습), 수렴 후 α=0.95(거의 고정)
      수렴 판정: 연속 3회 변화율 < 1% → speed_scale_converged = True
      변동성 가드: 표준편차/평균(CV) > 0.4 → 불안정 구간으로 갱신 스킵

  _speed_samples: maxlen=6000 (10분 @ 10fps)
    avg_speed_kph > 0인 프레임마다 (속도, 단조 시각) 튜플 추가.
    calibrate_from_its()와 _inject_its_speed()의 our_avg_kph 계산에 공통 사용.

  * 과속 기준: 카메라 전환 시 nodelink DB에서 해당 도로 max_spd 자동 적용.
    DB 없거나 정보 없으면 config.SPEED_LIMIT_KPH(기본 120) 사용.

  * 좌표 변환 기준점: bbox 기하학적 중심(center_px)이 아닌 바닥 중심
    (bottom center = cx, xyxy[3])을 호모그래피에 입력.
    center_px는 지도 마커 표시용으로만 별도 유지.

  VehicleState 주요 필드:
    track_id, class_name, bbox_xyxy, center_px
    lat, lon, x_m, y_m          (GPS + 미터 좌표)
    direction                    "In" / "Out" / "Unknown" (LineZone 교차 기반)
    speed_kph, is_speeding       (속도 및 과속 여부)
    dwell_frames, is_bottleneck, is_parked


[ transform.py ]
  PerspectiveTransformer — 픽셀 좌표 → GPS/미터 변환

  원리: 카메라 보정 시 수집한 4쌍의 (픽셀, GPS) 대응점으로
        OpenCV findHomography()를 통해 Homography 행렬 산출.
        이 행렬이 카메라 높이·각도·기울기를 암묵적으로 인코딩함.

  보정 데이터 없을 때: config.py의 기본 격자 좌표 사용 (부정확)
  보정 후:            calibration_data.json 로드 → 정밀 변환

  pixel_to_gps(cx, cy)   → (lat, lon)
    수동 보정 시: H_gps 직접 적용 (단일 호모그래피)
    미보정 + road_pts 설정 시: _pixel_to_gps_curved() 시도 → 실패 시 H_gps 폴백
  pixel_to_meter(cx, cy) → (x_m, y_m)  (속도 계산 보조)
  is_calibrated (property)
    - update_from_calibration() 호출 후 True
    - update_gps_center() (근사) 호출 후 False
    - auto_calibrate_from_frame() 성공 후에도 False 유지 (수동 4점 보정만 True)
    - camera_ready WS 메시지에 포함 → 프론트엔드 미보정 표시에 활용
  update_from_calibration(data): 보정 데이터 적용 + 행렬 재계산
    - _H_gps: 픽셀→GPS 호모그래피 재계산
    - _H_meter: GPS 4점에서 로컬 ENU 미터 좌표 계산(_gps_pts_to_local_meters)
                → 픽셀→미터 호모그래피도 함께 재계산 (두 행렬 일관성 보장)
    * 보정 전: _H_meter는 config.py 기본 격자값 사용 → 속도 오차 큼
    * 보정 후: 실측 GPS 기반 미터 좌표로 _H_meter 갱신 → 거리 계산 정확도 향상

  set_road_corridor(road_pts, snap_along_m):
    nodelink에서 받은 도로 중심선 포인트를 설정.
    road_pts: [[lat,lon],...] F→T 순서 (get_road_snap의 road_pts)
    snap_along_m: road_pts[0]에서 snap(카메라 도로 투영점)까지의 거리(m)
    설정 후 _pixel_to_gps_curved()가 활성화됨.
    main.py의 switch_camera 시 road_snap 결과로 호출.

  _pixel_to_gps_curved(u, v) → (lat, lon) | None:
    2단계 곡선 GPS 변환.
    Stage 1: pixel → (x_m, y_m) via H_meter  [평면 ENU 근사]
    Stage 2: (x_m, y_m) → snap 기준 ENU 변위 (dx, dy)
             → 도로 축 분해: d_along (도로 방향), d_lateral (횡방향)
             → target_arc = snap_along_m + d_along
             → road_pts에서 target_arc 위치 보간 → center GPS
             → 로컬 도로 방위로 d_lateral 적용 → 최종 GPS
    효과: 단일 호모그래피의 직선 투영 대비 도로 곡선을 따른 GPS 계산.
    None 반환 시(road_pts 미설정 등) H_gps 폴백.

  _snap_meter_x, _snap_meter_y:
    snap 좌표를 H_meter 좌표계(ENU 미터)로 표현한 값.
    _pixel_to_gps_curved에서 snap 기준 변위 계산에 사용.
    update_gps_center()와 auto_calibrate_from_frame() 모두 이 값을 갱신.
    (gps_pts[0] = TL 코너를 원점으로 하는 ENU 미터 좌표)

  auto_calibrate_from_frame(frame, bearing_deg, road_width_m) → bool
    수동 보정 없이도 차선 감지로 호모그래피를 자동 추정.
    성공 시 True 반환(+로그), 차선 미감지 시 False(기존 행렬 유지).

    처리 단계:
      1. Canny 엣지 검출
      2. HoughLinesP로 직선 세그먼트 추출
      3. 각도 필터링: 15°~75° (도로 사선) 만 통과, 수평·수직 제거
      4. 소실점(VP) 추정: 필터링된 직선 쌍의 교점 중위수
         - _line_intersection(l1, l2): 행렬식으로 교점 계산
         - VP 유효 범위: 프레임 상단 절반 이내
      5. 카메라 피치 역산:
           fy = 프레임 높이 × 1.2  (35mm 등가 환산 추정)
           pitch_deg = atan2(h/2 − vp_y, fy)  (단위: 도)
      6. 도로 경계 검출: 세그먼트 끝점 중 VP보다 아래인 점들로
         좌/우 경계 X 좌표 추정 → road_px_w 계산
      7. 핀홀 카메라 모델로 카메라 높이 역산:
           vfov_half = atan2(h/2, fy)  (수직 화각 절반)
           d_near = road_width_m × fy / road_px_w   (근거리 도로까지 거리)
           cam_h  = d_near × tan(pitch_deg + vfov_half)
           cam_h 클램핑: 3.0 m ~ 40.0 m
      8. near_m / far_m 계산 (카메라로부터 도로 영역 전·후 거리):
           near_m = d_near (클램핑: 3 ~ 30 m)
           far_m  = cam_h / tan(pitch_deg − vfov_half)  (클램핑: ≤300 m)
      9. 픽셀 좌표 사다리꼴 구성:
           하단(근거리): 전체 폭 하단 1/3
           상단(원거리): VP 근방 좁은 영역
      10. bearing_deg + near_m/far_m/road_width_m/2 로 GPS 코너 4점 계산
      11. findHomography() → _H_gps, _H_meter 동시 갱신

  FastAPI 앱의 진입점. 모든 엔드포인트와 백그라운드 루프 포함.

  REST 엔드포인트:

                            → speed_scale.json에서 카메라별 보정 계수 복원
                            → 수동 보정 없는 카메라: 노드링크 차선 수로 도로 폭 계산

                              (road_name, road_lanes, road_max_spd, road_bearing,
                               name_bearing 포함)
    GET  /nodelink/nodes  보정 모드용 주변 노드 목록 반환 (GPS 스냅핑용)
    GET  /cctv-refresh    HLS NETWORK_ERROR 시 신선한 URL 반환

  FOV polygon 계산 함수 3종:

  computeFovPolygon(lat, lon, headingDeg, fovDeg=70, distM=90, nearM=15):
    heading 방향으로 nearM~distM 범위, ±fovDeg/2 각도의 사다리꼴 폴리곤.
    road_pts가 없을 때 fallback으로 사용.

  computeRoadCorridorPolygon(roadPts, snapAlongM, headingDeg, nearM, farM, halfWidthM):
    nodelink shape_pts 폴리라인을 따라 도로 곡선을 반영하는 polygon.
    처리:
      1. roadPts를 따라 누적 거리(cumDist) 계산
      2. snapAlongM에 가장 가까운 로컬 도로 방위 계산
      3. headingDeg vs 로컬 방위 비교 → fwdIsFT(카메라가 F→T 방향을 보는지) 판별
      4. nearDist/farDist를 F→T 누적 거리계로 변환
         (fwdIsFT=true: snap+nearM ~ snap+farM, false: snap-farM ~ snap-nearM)
      5. startD ~ endD 구간의 도로 중심선 점 수집 (보간 포함)
      6. 각 점에서 도로 방향에 수직으로 ±halfWidthM 오프셋 적용
      7. 오른쪽 edge + 왼쪽 edge(역순) + 닫기로 polygon 구성
    반환 null 시 computeCalibPolygon으로 폴백.
    곡선 도로에서 직선 근사 FOV 대비 실제 도로 형태와 일치.

  computeCalibPolygon(lat, lon, headingDeg, nearM, farM, halfWidthM):
    자동 캘리브레이션 후 사용하는 직사각형 polygon.
    transform.py의 GPS 코너 계산과 동일 방식 (bearing 방향으로 회전한 직사각형).
    road_pts 없거나 computeRoadCorridorPolygon이 null 반환 시 사용.

  polygon 우선순위 (App.jsx에서 결정):
    1. computeRoadCorridorPolygon (road_pts 있을 때)
    2. computeCalibPolygon (auto_calibrated 메시지 수신 시)
    3. computeFovPolygon (그 외 기본 사다리꼴)

[ 속도 계산 로직 ]
  4단계 방어 구조로 순간 속도 폭등(스파이크)을 차단.

  ─── 단계 1: 중복 위치 스킵 ───
    YOLO_DETECT_INTERVAL=3으로 매 3프레임 중 1회만 YOLO 실행.
    비탐지 프레임은 _last_tracks(직전 결과)를 그대로 반환 → 같은 x_m, y_m.
    동일 위치는 _pos_window에 추가하지 않음 (Fix A).
    효과: 윈도우에는 YOLO 실제 탐지 시점 데이터만 누적 (~10fps).
          비탐지 프레임이 동일 좌표를 18번 채워 속도를 0으로 만드는 문제 방지.

  ─── 단계 2: 물리 기반 점프 감지 ───
    새 위치를 _pos_window에 추가하기 전, 직전 항목과의 변위를 검사.
      dt > 2.0초     → 이력이 너무 오래됨, 윈도우 초기화
      dist_m > (MAX_REASONABLE_KPH/3.6/speed_scale) × dt × 1.5
                     → 텔레포트(잘못된 ID 매칭, OcSort Kalman 이탈), 윈도우 초기화
    초기화 후 속도=0, 다음 2프레임 동안 윈도우 재구축.
    효과: IDStabilizer 오매칭, bbox 바닥 좌표 큰 변동 등으로 발생하는
          100 km/h 스파이크를 윈도우에 들어오기 전에 차단.

  ─── 단계 3: OLS 회귀 속도 추정 ───
    슬라이딩 윈도우(maxlen=18): (x_m, y_m, timestamp) 이력 유지.
    윈도우 전체 변위 < SPEED_JITTER_THRESHOLD_M(0.5m) → speed=0 (지터 제거).
    x(t), y(t) 각각에 대해 최소제곱법(OLS)으로 기울기(속도 성분) 추정.
      vx = d(x)/dt,  vy = d(y)/dt  (m/s)
      kph = sqrt(vx² + vy²) × 3.6
    kph × speed_scale > MAX_REASONABLE_KPH(120) → 이상치 스킵 (이번 프레임 유지).
    OLS 회귀는 단순 (newest - oldest)/dt 대비 내부 이상값에 강인함.

  ─── 단계 4: 자동 GPS 캘리브레이션 (auto_calibrate_from_frame) ───
    수동 4점 보정 없이 차선 감지만으로 호모그래피를 근사 추정.
    카메라 전환 직후 최대 5프레임 동안 시도 → 성공 시 중단.
    노드링크 차선 수로 도로 폭(m) 계산 → 핀홀 모델로 카메라 높이 역산.
    실패 시 기존 행렬(기본 격자) 유지 → 이후 ITS 보정으로 스케일 보완.

  ─── 단계 5: ITS 구간속도 자동 보정 (speed_scale) ───
    보정 후에도 호모그래피 스케일 오차로 ±50% 오차가 남을 수 있음.
    ITS API 5분 구간속도와 최근 10분 측정 평균을 비교해 speed_scale 학습.
      target = speed_scale × (ITS속도 / 측정평균)
      수렴 전: EMA α=0.7 (빠른 학습), 수렴 후: α=0.95 (거의 고정)
      수렴 판정: 연속 3회 변화율 < 1%
    수렴된 계수는 speed_scale.json에 카메라별 저장 → 재시작 후 즉시 복원.

  ─── 최종 출력 ───
    v.speed_kph = round(kph × speed_scale, 1)
    v.is_speeding = v.speed_kph > speed_limit_kph

  GC grace period: track_id가 GC_GRACE_FRAMES(90) 이상 미감지 시에만 삭제.
  3초 이내 재감지 시 이전 _pos_window 이력 연속성 유지.


[ 주차 차량 감지 ]
  방법 1 (dwell 기반): 연속 정지 300프레임(~10초) → is_parked=True
                        픽셀 위치를 _parked_positions에 등록

  방법 2 (위치 기반): 새로 탐지된 차량이 등록 위치 30px 이내
                       → track_id 무관하게 즉시 is_parked=True
                       (ID cycling에도 주차 상태 유지)

  주차 차량: 통계/경보에서 제외, 지도에 회색 마커로만 표시

10. 설계 결정 및 주의사항

  [트래커 인스턴스 공유 금지]
    live_loop과 /ws/detect가 같은 BoxMOT 트래커를 동시에 업데이트하면
    서로 다른 프레임 시퀀스가 섞여 트래킹이 완전히 망가짐.
    현재: /ws/detect 활성 시 live_loop의 track() 호출 스킵.
    /ws/detect 종료 시 reset_tracker() 호출로 상태 초기화.

  [Auto-ROI 비활성화]
    Canny edge detection 기반 자동 ROI 추정은 건물·교량 외곽을 도로로
    오인하여 차량 대부분을 필터링하는 문제가 있었음.
    카메라 전환 시 자동 추정·저장을 제거하고 수동 ROI만 적용.

  [외부 API 단건 응답 처리]
    ITS API는 결과가 1개일 때 list가 아닌 dict를 반환함.
    모든 API 응답에 isinstance(raw, dict) 분기 처리 적용.

  [HLS video 요소 정리]
    hls.destroy()는 HLS 내부 상태만 정리하고 video DOM 버퍼는 건드리지
    않음. 카메라 전환 시 video.src="" + video.load()로 직접 비워야
    이전 카메라 잔상이 남지 않음.

  [useEffect cleanup 스코프]
    useEffect 내 if 블록 안에서 const로 선언한 변수는
    if 블록 밖 cleanup 함수에서 접근 불가 (ReferenceError).
    cleanup에서 사용할 변수는 반드시 useEffect 최상위 스코프에
    let으로 선언해야 함.

  [4점 호모그래피 보정의 의미]
    픽셀→GPS 호모그래피는 단순 좌표 변환이 아니라 카메라 높이·각도·
    렌즈 왜곡을 모두 암묵적으로 인코딩함.
    정확한 속도 측정을 위해서는 실제 도로 위 측정 가능한 4개 지점으로
    보정하는 것이 필수. 기본값은 근사치라 오차가 큼.

  [거리별 속도 오차 문제]
    4점 호모그래피는 캘리브레이션 사각형 내부에서만 정확.
    프레임 상단(카메라에서 먼 차량)은 외삽(extrapolation) 구간으로
    GPS 좌표 오차가 누적되어 속도가 실제보다 낮게 측정되는 경향이 있음.
    캘리브레이션 후에도 체계적 ±50% 오차가 남을 수 있음.

    완화 전략:
      1. update_from_calibration()이 _H_gps와 _H_meter를 함께 갱신
         (두 행렬 일관성 확보, GPS 기반 haversine 거리 정확도 향상)
      2. 캘리브레이션 미수행 fallback: 사다리꼴 소스(상단 50% 폭 / 하단 100% 폭)로
         findHomography 입력 → 원근 변환 효과 부여 (Fix B)
         직사각형→직사각형은 선형 스케일로 퇴화해 근거리 과속/원거리 저속 발생
      3. OLS 회귀(18프레임) — 단발 오차 희석
      4. 자동 GPS 캘리브레이션(auto_calibrate_from_frame) — 차선 감지 기반 초기 근사
         수동 보정 없을 때 카메라 전환 직후 최대 5프레임 자동 시도
      5. ITS 구간속도 자동 보정(speed_scale) — 체계적 스케일 오차 장기 보정
         5분마다 ITS와 비교, 수렴 후 카메라별 JSON 저장 → 재시작 후 즉시 복원
      6. 캘리브레이션 포인트를 가까운 2점 + 먼 2점으로 배치하면
         호모그래피가 depth range를 더 잘 커버하여 오차 감소

  [중복 차량 ID 문제 — IDStabilizer _remap 누적]
    OcSort/ByteTrack이 탐지 공백 후 동일 차량에 새 raw_id를 부여할 때,
    IDStabilizer._find_lost()가 매칭에 성공해 _remap[new_raw_id] = stable_id를
    추가하면서 기존 _remap[old_raw_id] = stable_id를 제거하지 않으면,
    _remap에 같은 stable_id를 가리키는 항목이 프레임마다 축적됨.
    이후 트래커가 해당 raw_id들을 동시에 반환하는 순간 동일 display_id를 가진
    30+ 행이 한꺼번에 출력됨.
    해결: _find_lost() 성공 시 `_remap = {r:s for r,s in _remap.items() if s != stable}`
    로 stale 항목 일괄 제거 + 2-패스(기존 트랙 우선) 처리.

  [속도 스파이크 — OLS 창 내 이상값]
    OLS 회귀는 18개 샘플 중 1개만 이상값(텔레포트)이어도 기울기가 크게 왜곡.
    MAX_REASONABLE_KPH(120) 필터는 결과값 > 120만 차단해 100 km/h 허위값 통과.
    해결: _pos_window에 추가 전 물리 한계 검사.
          dist_m > (MAX_REASONABLE_KPH/3.6/speed_scale) × dt × 1.5 이면 창 초기화.
          IDStabilizer 오매칭, Kalman 이탈 등 모든 원인의 텔레포트를 일괄 차단.

  [OcSort + YOLO_DETECT_INTERVAL 충돌]
    OcSort 기본 min_hits=3: 동일 차량을 3연속 프레임에서 탐지해야 트랙 확정.
    YOLO_DETECT_INTERVAL=3이면 2프레임은 빈 배열 전달 → 트랙이 확정 전 계속 삭제.
    해결: OcSort(min_hits=1, max_age=BYTE_TRACK_BUFFER)로 초기화.
    ByteTrack도 동일 문제 → _patched_activate로 is_activated 즉시 True 설정.

    추가 문제: 빈 배열을 tracker.update()에 전달하면 트래커가 모든 트랙을
    'lost' 처리 → 다음 실제 탐지 시 새 raw_id 부여 → IDStabilizer 오매칭.
    해결: 연속 _YOLO_MISS_GRACE(=2)회 이내 빈 탐지 시 tracker.update()
    자체를 건너뛰고 _last_tracks 반환 → tracker 내부 상태 보존.

  [ITS 구간속도 자동 보정 설계]
    ITS API 응답 구조: JSON 형식은 {body: {items: [...]}} (직접 리스트).
    XML에서 변환된 형식은 {response: {body: {items: {item: [...]}}}}이므로
    _parse_traffic_items()에서 두 구조를 모두 처리.
    속도 필드명은 "speed" (avgSpd 아님).

    보정 계수 범위 클램핑: 0.3 ~ 5.0 (측정 불가 극단값 방지).
    최소 샘플 수 30개 이상, 측정 평균 3 km/h 이상 시에만 갱신
    (정차 상태에서 0÷0 오류 방지).

  [도로 방향 우선순위]
    카메라 FOV 방향 및 GPS 투영 축:
      1순위 name_bearing: CCTV 이름 내 방위 키워드 (남측/북측/(북)/동방향 등)
      2순위 road_bearing: nodelink DB F_NODE→T_NODE 방향
      3순위 없음: 투영 미적용 (차량 GPS 원본 그대로 지도 표시)

  [GPS 도로축 투영]
    _project_to_road_axis(): 차량 GPS를 road_bearing 방향 단위벡터에 투영.
    기준점으로 현재 프레임 차량들의 평균 위치를 사용 (카메라-도로 오프셋 흡수).
    횡방향 성분 제거 → 지도에서 차량이 도로 중심선 위에 정렬됨.
    In/Out 방향 판정은 투영 전 LineZone 교차 이벤트 기반이므로 영향 없음.

  [CCTV 이름 기반 도로 매칭]
    옆 도로 링크가 더 가깝게 조회되는 문제를 해결하기 위해
    "[국도 1호선]" 형태의 이름을 파싱해 nodelink 매칭에 활용.
    숫자(호선 번호) + 도로 종류 키워드 비교로 공백·표기 차이 허용.
    일치 링크 없으면 거리 기준 폴백.

  [Tkinter forward reference]
    model_setup.py의 _on_cuda_toggle은 fps_labels를 참조하는데,
    fps_labels가 그 이후 줄에서 채워짐. Python 클로저는 호출 시점에
    변수를 조회(late binding)하므로, UI 렌더링 완료 후에만 callback이
    실행되어 안전함.

  [ITS 10분 창 vs 5분 집계 — 타이밍 정렬 문제]
    ITS API는 고정 벽시계 간격(:00, :05, :10…)으로 5분 구간속도를 생성.
    서버 부팅 시각은 임의이므로 _its_speed_poll_loop 폴링 주기(:00)와
    ITS 갱신 경계(:05) 사이에 최대 5분 오프셋이 발생.
    5분 창을 쓰면 ITS 집계 구간과 우리 측정 창이 전혀 겹치지 않을 수 있음.
    10분 창(window_s=600)을 쓰면 어떤 오프셋이어도 5분짜리 ITS 창이
    반드시 우리 창 내부에 완전히 포함됨 → 항상 유효한 비교 보장.
    대가: 속도 변화가 심한 도로에서는 10분 평균이 순간 ITS 값과 더 많이 다를 수 있음.
    완화: CV > 0.4(표준편차/평균 40%) 조건으로 불안정 구간은 갱신 자체를 스킵.

  [ITS 비교 표시 — our_avg_kph vs avg_speed_kph]
    기존 UI는 avg_speed_kph(프레임 스냅샷 평균)를 ITS 5분 집계와 비교했으나,
    두 값의 시간 단위가 달라 의미 없는 비교였음.
    수정: _inject_its_speed()에서 _speed_samples를 직접 읽어 10분 롤링 평균
    (our_avg_kph)를 계산해 페이로드에 포함.
    calibrate_from_its()와 동일한 데이터 소스·창 크기를 사용하므로
    UI 표시와 내부 보정 계수 학습이 동일한 기준으로 정렬됨.

  [자동 GPS 캘리브레이션 설계 결정]
    목표: 수동 4점 보정 없이도 합리적인 초기 호모그래피 제공.
    선택: 차선 마킹 Hough 검출 + 소실점 추정 + 핀홀 카메라 모델.
    한계:
      - fy = h×1.2 가정 (실제 초점거리 미지): 스케일 오차 ±20~30% 가능
      - 야간·눈·비·차량 밀집 등으로 차선 미감지 시 fallback으로 기본 격자 유지
      - is_calibrated = False 유지 → 프론트엔드 미보정 표시, 수동 보정 유도
    보완: 남은 스케일 오차는 ITS speed_scale 보정이 장기적으로 흡수.
    차선 폭 결정: nodelink road_rank → 101/102=3.5m, 103=3.25m, 기타=3.0m.
    MOCT 데이터는 단방향 기준 차선 수를 제공하므로 lanes×lane_w = 편도 도로 폭.

  [양방향 도로 중심 보정 설계 결정]
    문제: 노드링크는 왕복 도로를 방향별 단방향 링크 2개로 저장.
          get_road_snap()이 반환하는 snap은 한쪽 carriageway 중심에 위치.
          → FOV polygon이 도로 한쪽으로 치우쳐 나타남.
    해결: _find_reverse_link()로 같은 도로 역방향 링크 탐색
          → _snap_for_link()로 역방향 snap 계산
          → 두 snap 평균 = 실제 도로 중심
          → road_pts도 동일 횡방향 오프셋으로 이동
    조건: 두 snap 간 거리 2~40m. 미만이면 동일 링크 중복, 초과이면 다른 도로.

  [역방향 링크 bearing 비교 시 local_bearing 사용 불가]
    문제: 곡선 도로에서 snap이 속한 세그먼트의 local_bearing(로컬 방위)이
          전체 링크의 bearing_deg(F→T 전체 방위)와 40~60° 차이 발생.
          local_bearing으로 역방향 링크 ±60° 조건 검사 시 매칭 실패.
          → 같은 도로의 일부 카메라는 양방향 중심을 찾고 일부는 못 찾는 불일관성.
    해결: _find_reverse_link()에 link["bearing_deg"](전체 F→T 방위)를 전달.
          같은 도로 같은 방향의 모든 snap에서 일관된 역방향 링크 탐색.

  [링크 경계 근방 snap → 정사각형 polygon 문제]
    문제: _road_corridor_pts()는 현재 링크 shape_pts 안에서만 동작.
          snap이 링크 앞쪽 경계 20m 지점이면 backward walk가 20m만 커버.
          nearDist=snap+nearM, farDist=snap-farM → 유효 corridor=0~5m → 정사각형 polygon.
    해결: _extend_pts_with_adjacent()로 shape_pts를 앞뒤 인접 링크로 확장 후 snap.
          snap이 경계 근방이어도 ±150m 범위의 충분한 corridor 구성.
    한계: 교차로(여러 링크 합류점)에서 인접 링크 선택이 모호할 수 있음.
          road_name + bearing ±60° 필터로 최적 링크 선택, 실패 시 원본 pts 유지.

  [46.9% 직선 polygon의 원인]
    MOCT_LINK.shp의 46.9%는 shape_pts가 F_NODE와 T_NODE 2점만인 직선 세그먼트.
    shape_pts가 2점이면 _extend_pts_with_adjacent()로 인접 링크를 붙여도 전체가 직선이
    아닌 한 polygon은 꺾임 없이 나옴. 데이터 특성이지 코드 버그가 아님.
    곡선 shape_pts를 보유한 링크는 computeRoadCorridorPolygon이 실제 도로 곡률 반영.

  [CCTV 이름 도로명 파싱 — 대괄호 없는 형식]
    문제: ITS API cctvname 필드는 대부분 "국도1호선 성환IC" 형태로 대괄호 없음.
          기존 regex가 "[국도 N호선]" 대괄호 형식만 매칭 → 대부분의 카메라에서 None 반환.
          → road_name_hint 없이 거리 기준 폴백 → 인접 다른 도로 링크 선택 오류.
    해결: regex에 대괄호 없는 "국도\s*\d+\s*호선?" 패턴을 추가 alternation으로 추가.
          두 형식 모두 지원, 대괄호 형식이 우선.

  [calibration 및 속도 측정 정확도 한계]
    현재 자동 캘리브레이션(auto_calibrate_from_frame)의 실제 정확도:
      - fy = h×1.2 (35mm 환산 근사): 실제 초점거리와 달라 스케일 오차 ±20~30% 가능
      - road_width_m: nodelink lanes × 차선폭 기반 추정 (실측값 아님)
        → 차선 수 오류, 중앙분리대, 갓길 포함 여부에 따라 오차 발생
      - Hough 차선 검출 품질: 야간, 비, 차량 밀집 시 차선 미검출 → fallback
    ITS speed_scale 자동 보정이 스케일 오차를 장기적으로 흡수하지만:
      - ITS 구간속도 자체가 5분 평균으로 순간 오차 내포
      - 카메라가 담당하는 도로 구간과 ITS 구간이 정확히 일치하지 않을 수 있음
      - 수동 4점 보정 없이는 is_calibrated=False → 체계적 ±50% 오차 잔존 가능
    신뢰도 향상을 위한 권장 사항:
      1. 수동 4점 보정 (CalibrationMode) — 실측 도로 지점 4개로 호모그래피 직접 계산
      2. 보정 포인트: 근거리 2점 + 원거리 2점으로 depth range 커버
      3. ITS speed_scale 수렴 대기 (연속 3회 변화율 < 1%)
