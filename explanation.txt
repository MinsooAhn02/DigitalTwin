이 문서는 프로젝트를 처음 접하는 사람도 전체 구조와 동작 방식을
이해할 수 있도록 작성된 기술 설명서입니다.


================================================================
1. 프로젝트 개요
================================================================

실제 도로의 CCTV 영상을 AI로 분석하여, 지도 위에 차량 위치·속도·
교통 흐름을 실시간으로 시각화하는 교통 디지털 트윈 시스템입니다.

핵심 기능:
  - ITS(국가교통정보센터) OpenAPI에서 실제 CCTV HLS 스트림 수신
  - YOLOv8 + BoxMOT 기반 차량 탐지 및 다중 객체 추적
  - 픽셀 좌표 → GPS 좌표 변환 (4점 호모그래피 보정)
  - deck.gl 지도 위 차량 마커, 속도, LOS, 경보 실시간 표시
  - ROI(관심 영역) 설정, 카메라 방향 보정 UI

3. 기술 스택
================================================================

[ 백엔드 — Python 3.11 ]
  웹 서버       FastAPI + Uvicorn (비동기 REST + WebSocket)
  AI 탐지       ultralytics YOLOv8x / YOLOv8s
                TensorRT FP16 (.engine) — GPU 가속 추론
                torch 2.6.0+cu124 (CUDA 12.4)
  차량 추적     BoxMOT — BotSort / ByteTrack / OcSort / DeepOcSort
                (TRACKER_TIER 환경변수로 선택, 기본 auto)
  영상 처리     OpenCV-Python (cv2), FFmpeg (HLS 디코딩)
  좌표 변환     numpy (homography 행렬 연산)
  HTTP          httpx (비동기 ITS API 호출)
  설정          python-dotenv, pydantic

[ 프론트엔드 — JavaScript (ES2022 + JSX) ]
  빌드          Vite + React 18 (함수형 컴포넌트 + Hooks)
  지도          deck.gl (ScatterplotLayer, TextLayer, PathLayer,
                         PolygonLayer, IconLayer)
                react-map-gl + MapLibre GL (벡터 타일)
                Carto Dark Matter 스타일 테마
  HLS 재생      hls.js (브라우저에서 m3u8 재생)
  차트          Recharts (파이 차트, 레이디얼 바)
  통신          브라우저 네이티브 WebSocket, fetch API
  상태 관리     useState / useReducer / useRef / useCallback / useMemo

[ 외부 서비스 ]
  ITS OpenAPI   https://openapi.its.go.kr:9443/cctvInfo
                현재 지도 뷰포트 bbox 기준 CCTV 목록(위치 + HLS URL) 조회
                HLS URL은 만료 토큰 포함 → 자동 갱신 로직 적용

5. 백엔드 모듈 상세
================================================================

[ config.py ]
  모든 하드코딩 상수를 한 곳에서 관리합니다.
  RUNTIME_PROFILE은 model_setup.py GUI 실행 후 .runtime_profile.json에
  저장되며, 서버 시작 시 이 파일을 읽어 동적으로 오버라이드합니다.

  주요 설정:
    YOLO_MODEL              사용할 모델 파일 (.pt 또는 .engine)
    YOLO_CONF / YOLO_IOU    탐지 임계값 (0.25 / 0.45)
    VEHICLE_CLASSES         탐지 대상 클래스 {2:car, 3:motorcycle, 5:bus, 7:truck}
    TRACKER_TIER            트래커 선택 (auto/cpu/low/medium/high)
    SPEED_LIMIT_KPH         과속 기준 (기본 120)
    SPEED_SMOOTHING_ALPHA   EMA 평활화 계수 (0.15 — 낮을수록 스파이크 잔류 감소)
    MAX_REASONABLE_KPH      이상치 제거 상한 (120 — 도로 제한속도 기준)
    GC_GRACE_FRAMES         트랙 GC 유예 프레임 수 (30 — 약 1초간 ID 보존)
    SPEED_WINDOW_FRAMES     슬라이딩 윈도우 속도 계산 이력 크기 (8프레임)
    YOLO_DETECT_INTERVAL    YOLO 추론 주기 (3 — 3프레임 중 1회만 추론)
    BOTTLENECK_DWELL_FRAMES 병목 판정 정지 프레임 (150 = 5초)
    PARKED_FRAMES_THRESHOLD 주차 판정 정지 프레임 (300 = 10초)
    PARKED_POSITION_RADIUS_PX  주차 위치 메모리 반경 (30px)


[ detector.py ]
  VehicleDetector
    - YOLO 모델 로딩: .pt (PyTorch) 또는 .engine (TensorRT FP16) 자동 선택
    - detect(frame): 단순 YOLO 추론 (tracker_id 없음)
    - track(frame): BoxMOT 트래커 연동 추론
        1. YOLO_DETECT_INTERVAL(=3)마다 YOLO predict 실행,
           나머지 프레임은 직전 탐지 결과 재사용 (YOLO 부하 2/3 절감)
        2. CUDA 환경에서 half=True(FP16) 추론 (약 30~50% 속도 향상)
        3. ROI 마스킹 (roi_manager에서 불러온 폴리곤 적용)
        4. tracker.update(bboxes, frame) → track_id 할당/유지 (Kalman 예측 포함)
        5. supervision.Detections 객체 반환
    - set_roi(polygon): ROI 폴리곤 업데이트
    - reset_tracker(): 카메라 전환 시 트래커 내부 상태 초기화
    - threading.Lock: live_loop + ws/detect 동시 접근 직렬화

  VideoStream
    - switch_to(url): OpenCV(FFmpeg 백엔드)로 HLS 스트림 전환
    - read_frame(): 프레임 읽기 (frame_id, ndarray)
    - reconnect(): 스트림 끊김 자동 재연결


[ tracker.py ]
  VehicleTracker
    - LineZone 교차 판정으로 In/Out 카운팅
    - ByteTrack은 제거됨 — 현재 BoxMOT이 tracker.py 외부에서 처리
    - update(detections, frame_wh): supervision LineZone 결과 반환


[ analytics.py ]
  TrafficAnalytics — 매 프레임마다 호출되는 교통 지표 계산 엔진

  입력: VehicleState 리스트 (track_id, class_name, center_px, lat, lon 등)
  출력: FrameAnalytics (vehicles, vehicle_count, avg_speed_kph, los_grade, ...)

  처리 순서:
    1. _gc(active): 미감지 트랙 정리 (30프레임 grace period 후 삭제)
    2. _is_near_parked(): 등록된 주차 위치 30px 이내 → 즉시 is_parked
    3. _speed(): 3단계 안정화 속도 계산
       - 단계A (bbox 센터 EMA): 픽셀 center_px를 EMA(α=0.15)로 스무딩
         → YOLO bbox 지터(10~20px 튐)를 흡수
       - 단계B (슬라이딩 윈도우): 최근 8프레임 (x_m, y_m, ts) 이력 유지
         가장 오래된 위치~현재 위치 총 이동거리 / 총 시간 → raw_kph
         단발 spike를 희석, 1프레임 노이즈 증폭 방지
       - dist < 0.5m → speed=0 (지터 제거)
       - raw_kph > 120 → 이상치 스킵 (EMA 유지)
       - EMA(α=0.15) 평활화 → speed_kph 확정
    4. _dwell_update(): 정지 프레임 누적 → 병목/주차 판정
    5. 주차 차량 제외 후 통계 집계 (avg_speed, LOS, class_counts)

  * 좌표 변환 기준점: bbox 기하학적 중심(center_px)이 아닌 바닥 중심
    (bottom center = cx, xyxy[3])을 호모그래피에 입력.
    호모그래피는 도로 평면 기준으로 캘리브레이션되므로
    차량이 실제로 닿는 지면 접촉점을 사용해야 거리·속도 오차가 최소화됨.
    center_px는 지도 마커 표시용으로만 별도 유지.

  VehicleState 주요 필드:
    track_id, class_name, bbox_xyxy, center_px
    lat, lon, x_m, y_m       (GPS + 미터 좌표)
    speed_kph, is_speeding    (속도 및 과속 여부)
    dwell_frames, is_bottleneck, is_parked


[ transform.py ]
  PerspectiveTransformer — 픽셀 좌표 → GPS/미터 변환

  원리: 카메라 보정 시 수집한 4쌍의 (픽셀, GPS) 대응점으로
        OpenCV findHomography()를 통해 Homography 행렬 산출.
        이 행렬이 카메라 높이·각도·기울기를 암묵적으로 인코딩함.

  보정 데이터 없을 때: config.py의 기본 격자 좌표 사용 (부정확)
  보정 후:            calibration_data.json 로드 → 정밀 변환

  pixel_to_gps(cx, cy)   → (lat, lon)
  pixel_to_meter(cx, cy) → (x_m, y_m)  (속도 계산 보조)
  update_from_calibration(data): 보정 데이터 적용 + 행렬 재계산
    - _H_gps: 픽셀→GPS 호모그래피 재계산
    - _H_meter: GPS 4점에서 로컬 ENU 미터 좌표 계산(_gps_pts_to_local_meters)
                → 픽셀→미터 호모그래피도 함께 재계산 (두 행렬 일관성 보장)
    * 보정 전: _H_meter는 config.py 기본 격자값 사용 → 속도 오차 큼
    * 보정 후: 실측 GPS 기반 미터 좌표로 _H_meter 갱신 → 거리 계산 정확도 향상


[ roi_manager.py ]
  ROI(관심 영역) 저장·로드 관리

  - 정규화 좌표 ([0,1] 범위) 로 저장 → 해상도 변경에 독립적
  - roi_config.json에 카메라 URL 기반 키로 저장
  - get_roi(cctvurl): 저장된 ROI 폴리곤 반환 (없으면 None)
  - save_roi(cctvurl, polygon): 폴리곤 저장
  - delete_roi(cctvurl): ROI 삭제


[ model_setup.py ]
  최초 실행 시 Tkinter GUI로 모델 선택 대화상자 표시.

  선택 항목:
    - YOLO 모델 (yolov8x / yolov8s / 커스텀)
    - 성능 프로파일 (quality / balanced / performance)
    - CUDA 사용 여부 (TensorRT 활성화)
    - 트래커 티어 (auto / cpu / low / medium / high)

  결과를 .yolo_model과 .runtime_profile.json에 저장.
  CUDA 체크박스 토글 시 각 모델의 FPS 수치가 실시간으로 갱신됨.


[ main.py ]
  FastAPI 앱의 진입점. 모든 엔드포인트와 백그라운드 루프 포함.

  REST 엔드포인트:
    GET  /cctvs           ITS API에서 뷰포트 내 CCTV 목록 조회 (TTL 5분 캐시)
                            cctvname 필드에 _korname_to_en() 적용:
                            "국도 3호선 상행" → "국도 3호선 상행 (Nat'l Rt.3 NB↑)"
    POST /switch-camera   카메라 전환:
                            → _camera_queue push (live_loop 스트림 전환)
                            → detector.reset_tracker()
                            → transform.update_gps_center()
                            → analytics.reset()
                            → "camera_ready" WS 신호 발송
    GET  /cctv-refresh    HLS NETWORK_ERROR 시 신선한 URL 반환
    GET  /runtime-config  현재 캡처 설정 반환 (프론트엔드 동기화용)
    GET/POST/DELETE /roi  ROI 폴리곤 관리
    GET/POST/DELETE /calibration  보정 데이터 관리
    GET  /health          서버 상태

  WebSocket:
    WS /ws          JSON 브로드캐스트 구독 (FrameAnalytics 데이터)
    WS /ws/detect   JPEG 수신 → 추론 → 어노테이션 JPEG 반환
                    수신 시 live_loop의 track() 호출을 스킵 (동시성 충돌 방지)

  백그라운드:
    live_loop       OpenCV로 HLS 스트림 읽기 → 탐지 → 브로드캐스트
    hls_refresh_loop  30분마다 ITS API 재호출 → URL 갱신

8. AI 탐지·추적 파이프라인 상세
================================================================

[ YOLO 모델 ]
  yolov8x.engine  TensorRT FP16, ~9.6ms/frame (RTX 4070 기준)
  yolov8s.engine  TensorRT FP16, ~3ms/frame
  yolov8x.pt      PyTorch GPU, ~33ms/frame (TensorRT 없을 때)

  TensorRT .engine 파일은 실행 GPU에서 직접 변환해야 함.
  다른 GPU에서 만든 파일은 사용 불가.


[ BoxMOT 트래커 티어 ]
  TRACKER_TIER 환경변수 (config.py) 또는 model_setup GUI로 선택:

    auto    VRAM 크기 자동 감지 → 적합한 티어 선택
    cpu     ByteTrack  — ReID 없음, VRAM 불필요, 가장 빠름
    low     OcSort     — ReID 없음, 가림 처리 강함
    medium  BotSort    — appearance ReID, VRAM 6~8GB 권장
    high    DeepOcSort — appearance ReID, VRAM 8GB+ 권장

  ReID(Re-Identification): 차량이 잠시 화면에서 사라져도
  외형 특징으로 동일 track_id 재할당 → ID 끊김 최소화.


[ 속도 계산 로직 ]
  속도 불안정의 3가지 원인을 각각 차단하는 구조로 설계됨.

  원인 1 — Track ID 재탄생:
    YOLO 탐지 누락·가림 시 BoxMOT이 새 track_id를 부여 → 위치 이력 없음
    → 재탄생 직후 프레임에서 dt가 매우 짧거나 이전 위치 오차가 커 속도 폭등.
    대응: GC_GRACE_FRAMES=30 (약 1초) 동안 미감지 track_id의 이력·EMA 보존.
          차량이 잠깐 가려져도 동일 ID로 복귀 가능성 대폭 증가.

  원인 2 — YOLO bbox 픽셀 지터:
    동일 차량이라도 bbox 중심이 매 프레임 10~20px씩 불규칙 이동.
    640px = 20m 스케일에서 0.3~0.6m/frame → 30fps 기준 40~70 km/h 유령 속도.
    대응: center_px를 EMA(α=0.15)로 먼저 스무딩 (bbox 센터 EMA).

  원인 3 — 1프레임 차이 속도 계산:
    N→N+1 프레임 이동거리만으로 속도 계산 시 순간 bbox 튐이 그대로 반영.
    대응: 슬라이딩 윈도우(8프레임) — 가장 오래된 위치~현재 위치 총 이동거리 /
          총 경과 시간으로 평균 속도 산출. 단발 튐이 8배 희석됨.

  계산 순서:
    1. main.py에서 bbox 바닥 중심(gx=cx, gy=xyxy[3])으로 호모그래피 변환
       → (x_m, y_m) 획득 (도로 평면 기준 정확한 실세계 좌표)
    2. center_px EMA 스무딩 (α=0.15) — bbox 픽셀 지터 흡수
    3. (x_m, y_m, timestamp) 을 슬라이딩 윈도우(maxlen=8)에 push
    4. 윈도우[0] → 윈도우[-1] 누적 dist_m / dt → raw_kph
    5. dist_m < 0.5m → speed=0 (지터 제거)
    6. raw_kph > 120 → 이상치 스킵, 이전 EMA 유지
    7. EMA: speed = 0.15 × raw + 0.85 × prev_ema (최종 평활화)

  GC grace period: track_id가 30프레임 이상 미감지 시에만 삭제
  → 1초 이내 재감지 시 이전 위치/속도/윈도우 이력 연속성 유지


[ 주차 차량 감지 ]
  방법 1 (dwell 기반): 연속 정지 300프레임(~10초) → is_parked=True
                        픽셀 위치를 _parked_positions에 등록

  방법 2 (위치 기반): 새로 탐지된 차량이 등록 위치 30px 이내
                       → track_id 무관하게 즉시 is_parked=True
                       (ID cycling에도 주차 상태 유지)

  주차 차량: 통계/경보에서 제외, 지도에 회색 마커로만 표시

9. 성능 수치 (RTX 4070 Laptop 기준)
================================================================

  파이프라인 구간별 소요 시간
    캔버스 캡처 + JPEG 인코딩    ~8ms
    WebSocket 왕복 (로컬호스트)  ~3ms
    JPEG 디코딩 (서버)           ~3ms
    YOLO + BoxMOT 추론           ~9.6ms (yolov8x) / ~3ms (yolov8s)
    어노테이션 + JPEG 인코딩     ~3ms
    ─────────────────────────────────────
    총 파이프라인                ~27ms  (~37fps) yolov8x
                                 ~20ms  (~50fps) yolov8s

  TensorRT FP16 전환 효과
    CPU(yolov8x)         ~500ms/frame
    GPU+PyTorch(yolov8x) ~33ms/frame
    TensorRT FP16(yolov8x) ~9.6ms/frame
    정확도 변화 없음 (FP16 오차 < 1%)

  YOLO_DETECT_INTERVAL=3 효과 (live_loop 기준)
    매 프레임 YOLO          ~9.6ms × 30fps = 288ms/초 GPU 점유
    3프레임 중 1회 YOLO     ~9.6ms × 10fps = 96ms/초 GPU 점유
    나머지 2프레임은 tracker Kalman 예측만 (~0.5ms) → GPU 여유 증가

  model.predict(half=True) 효과 (CUDA 환경)
    FP32 → FP16 전환: ~30~50% 추론 속도 향상 (.pt 모델 한정)
    TensorRT .engine은 이미 FP16이므로 변화 없음

  inFlight 제어 (maxInFlight=2)
    동시 전송 2프레임 제한 → 처리량 최대화 + 순서 보장

10. 설계 결정 및 주의사항
================================================================

  [트래커 인스턴스 공유 금지]
    live_loop과 /ws/detect가 같은 BoxMOT 트래커를 동시에 업데이트하면
    서로 다른 프레임 시퀀스가 섞여 트래킹이 완전히 망가짐.
    현재: /ws/detect 활성 시 live_loop의 track() 호출 스킵.
    /ws/detect 종료 시 reset_tracker() 호출로 상태 초기화.

  [Auto-ROI 비활성화]
    Canny edge detection 기반 자동 ROI 추정은 건물·교량 외곽을 도로로
    오인하여 차량 대부분을 필터링하는 문제가 있었음.
    카메라 전환 시 자동 추정·저장을 제거하고 수동 ROI만 적용.

  [외부 API 단건 응답 처리]
    ITS API는 결과가 1개일 때 list가 아닌 dict를 반환함.
    모든 API 응답에 isinstance(raw, dict) 분기 처리 적용.

  [HLS video 요소 정리]
    hls.destroy()는 HLS 내부 상태만 정리하고 video DOM 버퍼는 건드리지
    않음. 카메라 전환 시 video.src="" + video.load()로 직접 비워야
    이전 카메라 잔상이 남지 않음.

  [useEffect cleanup 스코프]
    useEffect 내 if 블록 안에서 const로 선언한 변수는
    if 블록 밖 cleanup 함수에서 접근 불가 (ReferenceError).
    cleanup에서 사용할 변수는 반드시 useEffect 최상위 스코프에
    let으로 선언해야 함.

  [4점 호모그래피 보정의 의미]
    픽셀→GPS 호모그래피는 단순 좌표 변환이 아니라 카메라 높이·각도·
    렌즈 왜곡을 모두 암묵적으로 인코딩함.
    정확한 속도 측정을 위해서는 실제 도로 위 측정 가능한 4개 지점으로
    보정하는 것이 필수. 기본값은 근사치라 오차가 큼.

  [거리별 속도 오차 문제]
    4점 호모그래피는 캘리브레이션 사각형 내부에서만 정확.
    프레임 상단(카메라에서 먼 차량)은 외삽(extrapolation) 구간으로
    GPS 좌표 오차가 누적되어 속도가 실제보다 낮게 측정되는 경향이 있음.

    완화 전략:
      1. update_from_calibration()이 _H_gps와 _H_meter를 함께 갱신
         (두 행렬 일관성 확보, GPS 기반 haversine 거리 정확도 향상)
      2. 슬라이딩 윈도우(8프레임) 평균 속도 + EMA(α=0.15) 이중 평활화
         — 단발 오차 희석 + 낮은 속도값 잔류 시간 단축
      3. 캘리브레이션 포인트를 가까운 2점 + 먼 2점으로 배치하면
         호모그래피가 depth range를 더 잘 커버하여 오차 감소

  [Tkinter forward reference]
    model_setup.py의 _on_cuda_toggle은 fps_labels를 참조하는데,
    fps_labels가 그 이후 줄에서 채워짐. Python 클로저는 호출 시점에
    변수를 조회(late binding)하므로, UI 렌더링 완료 후에만 callback이
    실행되어 안전함.
